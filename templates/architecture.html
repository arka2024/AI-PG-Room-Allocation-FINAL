Weighted Cosine Similarity
S(A, B) = Σ( wi · Pi,A · Pi,B ) / √( Σ( wi · Pi,A² ) · Σ( wi · Pi,B² ) )
Where wi = importance weight for dimension i, Pi = encoded feature value
Input Two user feature vectors + weight vector
Process Weighted dot product / magnitude product
Output Compatibility score 0–100% + ranked list

Phase 2

Database Design (Model-Driven)

Build the database schema based on ML needs. Store raw persona data, compute vectors dynamically — never store precomputed compatibility scores.

User Schema
// User Document { "profile": { name, age, gender, occupation, bio }, "location": { type: "Point", coordinates: [lon, lat], area_name: "district" }, "preferences": { budget, gender_pref, smoking, pets_allowed, veg_nonveg }, "persona_raw": { sleep_time, cleanliness_rating, noise_tolerance, introversion_score, cooking_freq, guest_freq, ... }, feature_vector: // computed dynamically }
Design Principles
Store raw persona data
Keep original values for re-encoding flexibility
Compute vectors dynamically
Allows weight tuning without data migration
Never store compatibility scores
Scores are O(n²) and stale on any profile update
Indexes
location → 2dsphere budget gender _id (hashed)

Phase 6

Geo Filtering Layer

Before similarity calculation, filter candidates by geography and preferences to reduce computation cost.

Pre-Filter Pipeline
District Filter

Haversine distance within configurable radius

Budget Range

min-max overlap between user preferences

Gender Preference

Mutual preference matching

ML Similarity Engine

Only filtered candidates pass to cosine similarity

Haversine Distance Function
d = 2r · arcsin( √( sin²(Δφ/2) + cos(φ₁) · cos(φ₂) · sin²(Δλ/2) ) )
r = Earth radius, φ = latitude, λ = longitude
Performance Benefit

Geo-filtering reduces candidate pool by ~80%, making ML computation feasible at scale.
